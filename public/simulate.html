Fire a lifecycle event as if it came from the HRIS — a demo/testing stand-in for a real HRIS webhook subscription, not something an end user would ever click in production.

Starter

A new joiner signs their offer. Core HR record is created and every downstream system provisions automatically.

02

Mover

An internal promotion or market transfer. Search for an existing employee, then set their new department.

03

Leaver

Last day confirmed. Access is revoked on schedule and data is retained per GDPR — nothing lingers, nothing breaks early.

— fire an event above, then check the homepage or Manage active users to see it —
How this is wired: Starter/Leaver call POST /api/dev/simulate-hris, which builds a realistic HRIS webhook payload from the name you enter, signs it, and feeds it through the same POST /api/webhooks/hris logic a real HRIS integration would hit. Mover works differently — it targets a real, already-existing employee (searched directly from the database) and calls POST /api/dev/simulate-mover, which updates their department and opens a new mover ticket for them right away. Nothing on any ticket advances on its own — head to Manage active users to step it through its pipeline by hand.
